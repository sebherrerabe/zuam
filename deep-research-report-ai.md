## MVP proposal

This MVP proposal turns the above evidence into a concrete module that is *invoke-only*, approval-oriented, and designed for ADHD usersÔÇÖ planning friction without making therapeutic claims.

**Invocation and interaction loop (invoke-only)**  
The user opens the agent chatbox and asks for help (e.g., ÔÇťPlan my afternoonÔÇŁ, ÔÇťConvert these notes into tasksÔÇŁ, ÔÇťMake space for a 45-minute focus blockÔÇŁ). The assistant responds with a **draft plan** that explicitly separates: (a) what it understood, (b) the proposed goal, (c) a small set of options, and (d) the exact actions it would take if approved. This aligns with mixed-initiative work showing humans refine goals and compare alternatives during planning, and with human-AI guidelines that emphasise making system behaviour legible and correction easy. ţłÇciteţłéturn13view10ţłéturn18view5ţłéturn13view8ţłü  

**Structured plan output for ADHD-friendly comprehension**  
Based on qualitative coping evidence (routines, cues, breaks, ÔÇťlow-level stressÔÇŁ techniques), the plan format should include: a ÔÇťfirst tiny stepÔÇŁ, built-in breaks, and an explicit ÔÇťstart cueÔÇŁ (timer, environmental cue, nudge), plus an optional ÔÇťif-then planÔÇŁ (ÔÇťIf I feel stuck, then open the smallest subtask and do 2 minutesÔÇŁ). **Review surface: layered, not chat-only**  
After drafting, the UI shows a multi-layer review surface:

- **Conversational summary** (what will happen)  
- **Checklist** (each action with a tick-box)  
- **Task diff** (before/after for edited items)  
- **Calendar timeline preview** (visual placement, conflicts, buffers)

**Approval and execution model**  
Execution happens only after approval. To reduce confirmation fatigue while keeping safety:

- The assistant requests **bundle approval** for low-risk task edits (create task, edit task text, move between lists).  
- It requests **explicit, high-salience approval** for higher-risk actions (archive/delete, marking many tasks complete at once, calendar moves that create conflicts).

**Action log + undo as first-class primitives**  
Every executed action is logged (what changed, when, why, and which prompt triggered it). Provide:

- **Undo** per action and per bundle  
- **Version history** for tasks/lists/focus blocks  
- **ÔÇťRevert planÔÇŁ** to rollback a multi-step execution

**Permission model and tool scoping**  
Default permissions should be minimal:

- Tasks: read/write within Zuam workspace (with recovery).  
- Calendar: read-only by default; write only to a dedicated focus-block calendar (or tagged blocks) unless the user opts in.  
- Notes/body content: metadata-only by default; explicit opt-in and scoping for full bodies.

- Start with **no long-term memory by default**.

**Tool-using architecture for safe execution**

1. **Planner** produces proposed actions and explains rationale.  
2. **Validator** checks tool schemas, invariants (no deletes without undo window, no calendar edits outside allowed calendars), and conflicts.  
3. **Preview renderer** generates diffs and timeline previews.  
4. **User approval gate** (risk-tiered).  
5. **Executor** runs commands with idempotency keys + transactional semantics.
